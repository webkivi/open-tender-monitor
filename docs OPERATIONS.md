# Эксплуатация

## Ежедневная проверка

1. Яндекс.Браузер открыт.
2. Расширение включено и не показывает ошибок.
3. Интервал равен 45 секундам.
4. Локальный сервис отвечает на `/api/status`.
5. Последний heartbeat не старше 120 секунд.
6. Telegram-бот доступен.

## Запуск и остановка

```bash
./scripts/start.sh
./scripts/stop.sh
```

Логи: `logs/open-tender-monitor.log`.

## После изменения расширения

1. Открыть `browser://extensions`.
2. Нажать круглую стрелку у `Open Tender Monitor: EIS RSS Bridge`.
3. Открыть параметры и нажать «Проверить».
4. Убедиться, что новый heartbeat появился в `/api/status`.

## Диагностика задержки

1. Взять `published_at`, `first_seen_at` и `notified_at` закупки.
2. Проверить историю `source_checks`.
3. Если heartbeat идёт регулярно, а закупка поздно появилась в RSS, задержка
   находится на стороне ЕИС.
4. Если heartbeat имеет разрывы, проверить браузер, сон компьютера и расширение.
5. Если монитор принял запись быстро, а Telegram задержался, проверить канал
   уведомлений и SMTP.

## Секреты

- `.env` — Telegram, SMTP и токен приёма.
- `yandex-eis-rss-bridge/private-config.js` — адрес сервиса и токен связи.

Эти файлы не отправляются в Git и не включаются в отчёты.

## Резервное копирование

Для восстановления нужны:

- исходный код;
- `.env`;
- `private-config.js`;
- `data/monitor.db`;
- список RSS из настроек расширения.

## Известные ограничения

- WSL IP может измениться после перезапуска.
- Браузерные alarms могут задерживаться при сне компьютера.
- SMTP пока не подтверждён как стабильный.
- Официальный RSS может публиковать запись позже самой карточки ЕИС.

## Очередь уведомлений

Новая релевантная закупка сначала сохраняется в SQLite, затем получает независимые
задания Telegram и email. Поэтому недоступность SMTP не замедляет срочное сообщение
в Telegram и не отменяет повторные попытки Telegram.

В `/api/status` поле `notifications` показывает количество заданий по состояниям:
`pending`, `sending`, `sent`, `failed`, `suppressed`.

`failed` означает шесть неудачных попыток одного канала. Сначала проверяется сеть и
настройки канала, затем задание можно безопасно повторно поставить в очередь отдельной
служебной командой. Автоматического обхода защит площадок или смены IP система не делает.
